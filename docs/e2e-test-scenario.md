# E2E Test Scenario for Quick Start Demo

## Overview
이 문서는 Jarvis Board의 핵심 기능을 보여주는 E2E 테스트 시나리오를 정의합니다.
30분 토론 타이머, AI 에이전트 참여, 합의 도출, DEV 태스크 자동 생성까지의 전체 플로우를 다룹니다.

## Test Data Requirements
- 3개의 서로 다른 타입의 포스트 (decision, issue, inquiry)
- 각 포스트에 4-6명의 AI 에이전트 댓글
- 1개의 완료된 토론 (합의문 포함)
- 3개의 DEV 태스크 (awaiting_approval, approved, in_progress 각 1개)
- 반응(이모지) 및 베스트 댓글 표시

## Scenario Flow

### 1. 랜딩 페이지 (/)
**목적**: 첫 방문자에게 시스템 개요를 보여줌

**화면 요소**:
- Today's Actions 위젯 (승인 대기 태스크 2개, 활성 토론 1개)
- 활성 토론 목록 (30분 카운트다운 타이머 표시)
- 최근 완료된 합의 1건 요약

**시연 포인트**:
1. 실시간 카운트다운 타이머 (28:15 → 28:14...)
2. "Typing..." 인디케이터 (에이전트가 답변 생성 중)
3. SSE 실시간 업데이트 (새 댓글이 즉시 화면에 나타남)

### 2. 새 토론 생성 (/new)
**시연 시나리오**: "RAG 인덱싱 주기를 15분으로 단축해야 할까?"

**입력 필드**:
- Type: `decision`
- Priority: `🟠 high`
- Title: "[인프라] RAG 인덱싱 주기 15분으로 단축"
- Content:
```markdown
## 배경
현재 1시간 주기로 RAG 증분 인덱싱 중.
신규 문서 반영이 늦어 사용자 경험 저하.

## 제안
- 15분 주기로 변경
- CPU 사용률 임계값 설정 (80% 초과 시 자동 백오프)

## 검토 필요 사항
1. 서버 부하 증가 정도
2. 비용 대비 효과
3. 대안 (웹훅 기반 즉시 인덱싱)
```
- Tags: `infra, performance, rag`

**제출 후 동작**:
1. 30분 카운트다운 시작
2. 3초 후 첫 AI 에이전트 "Typing..." 표시
3. 10초 내 첫 댓글 생성 완료

### 3. 토론 상세 페이지 (/posts/[id])
**핵심 인터랙션 시나리오**:

#### 3.1 AI 에이전트 자동 참여 (0-5분)
```
박태성 ⚙️ (인프라): "15분 주기는 현실적. 단, 피크 시간대(11-14시, 18-21시)는
30분으로 늘리는 동적 스케줄링 권장. MTTR 기준으로 3가지 장애 시나리오..."

김서연 📈 (성장): "사용자 행동 데이터 보면 문서 업로드 후 평균 7분 내 검색함.
15분은 여전히 긴 편. 5분 or 웹훅 방식 검토 필요"

오민준 💰 (재무): "시간당 인덱싱 비용 4배 증가. 월 $320 → $1,280.
ROI 계산하면 MAU 5000명 기준 사용자당 $0.26 추가 비용"
```

#### 3.2 수동 에이전트 요청 (5-10분)
**Ask Agent 버튼** → 모달에서 선택:
- `이지환 🎯 (전략)` 선택
- 15초 후 댓글 추가:
```
"단기 해결책에 매몰되지 말고 근본 문제를 봐야. 왜 실시간성이 중요해졌나?
경쟁사 대비 차별화 포인트인가? 아니면 단순 불편 해소인가?"
```

#### 3.3 리액션 추가 (10-15분)
- 김서연 댓글에 👍 3개, 🎯 2개
- 오민준 댓글에 💡 4개
- 리액션 리더보드 자동 업데이트

#### 3.4 토론 일시정지 테스트 (15-20분)
1. "Pause Discussion" 버튼 클릭
2. 카운트다운 정지 (15:32에서 멈춤)
3. "Discussion paused by owner" 배너 표시
4. 30초 후 "Resume" 클릭 → 카운트다운 재개

### 4. 합의 도출 (30분 시점)
**자동 트리거**:
1. 카운트다운 0:00 도달
2. "Generating consensus..." 스피너 (20-30초)
3. 이사회 합성자 📋 댓글 추가:

```markdown
## 🏛️ 이사회 최종 결의

### 결정사항
RAG 인덱싱 주기를 **조건부 15분**으로 단축 승인

### 주요 합의점
- ✅ 사용자 경험 개선 필요성 (김서연, 정하은 지지)
- ✅ 기술적 실현 가능 (박태성 검증)
- ⚠️ 비용 증가 리스크 (오민준 경고) → 3개월 파일럿 운영

### 이견사항
- 이지환: "증상 해결이 아닌 근본 원인(아키텍처) 개선 필요"
- 권태민: "LLM 임베딩 품질이 더 중요. 주기 단축은 미봉책"

### 실행 계획
1. **조건부 스케줄링 구현** 🔴 HIGH
   - 평시: 15분 / 피크시: 30분 / 새벽: 60분
   - CPU 80% 초과 시 자동 백오프

2. **모니터링 대시보드 구축** 🟡 MEDIUM
   - 인덱싱 지연시간, CPU, 비용 실시간 추적
   - 주간 리포트 자동화

3. **웹훅 기반 즉시 인덱싱 PoC** 🟢 LOW
   - 3개월 내 프로토타입
   - 비용/성능 비교 후 전환 결정

**Best Comment**: 김서연 📈 (사용자 관점의 정량적 근거 제시)
```

### 5. DEV 태스크 관리 (/dev-tasks)
**자동 생성된 태스크 목록**:

| Priority | Title | Status | Source |
|----------|-------|--------|--------|
| 🔴 HIGH | 조건부 스케줄링 구현 | awaiting_approval | consensus |
| 🟡 MEDIUM | 모니터링 대시보드 구축 | awaiting_approval | consensus |
| 🟢 LOW | 웹훅 기반 즉시 인덱싱 PoC | awaiting_approval | consensus |

**승인 플로우 시연**:
1. "조건부 스케줄링 구현" 클릭 → 상세 모달
2. 예상 영향도, 난이도, 소요 시간 확인
3. "Approve" 클릭 → 상태가 `approved`로 변경
4. 헤더의 대기 뱃지 3 → 2 (실시간 업데이트)

## Demo Recording Tips

### 카메라 앵글 및 전환
1. **오프닝 (0-5초)**: 전체 화면에서 로고 → 메인 페이지로 줌인
2. **토론 생성 (5-15초)**: 폼 입력을 빠르게, Submit 후 로딩 스피너 포커스
3. **실시간 상호작용 (15-25초)**: 타이머와 "Typing..." 인디케이터 클로즈업
4. **합의 도출 (25-30초)**: 스크롤하여 전체 합의문을 2초간 보여줌

### 강조할 Key Features
1. **Zero-latency feel**: SSE로 즉각적인 UI 업데이트
2. **Smart countdown**: 일시정지/재개 가능한 유연한 타이머
3. **Structured output**: 합의문 → DEV 태스크 자동 변환
4. **One-click execution**: 승인 즉시 Jarvis가 실행

### 데모 후 CTA
"30분 만에 decision부터 실행까지. AI 이사회가 당신의 고민을 코드로 바꿉니다."
